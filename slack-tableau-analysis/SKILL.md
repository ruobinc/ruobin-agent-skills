# Slack-Tableau 仮説駆動分析

Slack会話 → 仮説生成 → Tableauデータ検証 → Web補強 → Index付与 → レポート作成 → フロー図（任意） → Canvas共有の8ステップワークフロー。

**このワークフローは厳密に順番通り実行すること。ステップを飛ばさない。**

## ワークフロー

### Step 1: Slack情報収集 → 仮説生成

ユーザーの質問に関連するSlack会話を検索し、社内の議論・報告・懸念事項から仮説を導出する。

1. `slack_search_public_and_private`でキーワード検索（質問のテーマ + 関連用語を2-3パターン試す）
2. ヒットしたチャンネル・スレッドを`slack_read_channel` / `slack_read_thread`で深掘り
3. 収集した情報から**3-5個の仮説**を生成

仮説のフォーマット:

```
## 仮説一覧

1. **[仮説タイトル]** - [根拠となるSlack会話の要約]
   - 情報源: #channel名, @ユーザー名, 日付
   - Slack会話リンク: [メッセージのpermalink]
2. ...
```

Slack検索結果にはメッセージのpermalinkが含まれる。各仮説の根拠となった会話のリンクを必ず記載すること。複数の会話が根拠の場合は全てのリンクを列挙する。

ツール詳細: [references/slack-mcp-tools.md](references/slack-mcp-tools.md)

### Step 2: ユーザーに仮説を提示 → 選択確認

- 生成した仮説一覧をユーザーに提示
- `ask_user_input_v0` ツールで仮説の選択ウィジェットを表示する
- **ここで必ずレスポンスを終了し、ユーザーの返答を待つ**
- **ユーザーが仮説を選択・送信してから、はじめてStep 3に進む**
- Step 2のレスポンス内でStep 3以降のツール（Tableau MCP等）を一切呼び出さない

> ⚠️ HARD STOP: Step 2完了後は必ずターンを終了すること。ユーザーの次のメッセージを受け取るまでStep 3を開始しない。

### Step 3: Tableauデータ構造の確認

選択された仮説の検証に使えるデータを探索する。

1. `list-datasources`で利用可能なデータソース一覧を取得
2. 関連しそうなデータソースに対して`get-datasource-metadata`でフィールド構造を確認
3. 必要に応じて`list-views` / `list-workbooks`で既存ダッシュボードも確認
4. どのデータソース・フィールドで仮説を検証できるか整理

ツール詳細: [references/tableau-mcp-tools.md](references/tableau-mcp-tools.md)

### Step 4: Tableauデータ取得 → 分析

1. `query-datasource`で仮説検証に必要なデータを取得（フィルタ・集計を適切に指定）
2. 補完的に`get-view-data`で既存ビューのデータも取得
3. 取得データを分析し、仮説を支持/棄却する根拠を整理

分析時の注意:

- 数値は具体的に示す（「増加した」ではなく「前月比+15%」）
- 比較対象を明確にする（前期比、前年同期比、目標比等）
- 相関と因果を区別する

### Step 5: Web情報による補強（条件付き）

以下の場合にWeb検索を実行:

- 業界トレンドとの比較が有効な場合
- 外部要因（市場動向、競合、規制変更等）の影響が疑われる場合
- Tableauデータだけでは仮説の検証が不十分な場合

Web検索はClaude Desktopの組み込みWeb検索機能を使用する。

### Step 6: データソースIndex付与

Step 1〜5で使用した全データソースにIndexを付与する。Web補強の結果も含めて統合する。

フォーマット:

```
## データソースIndex

| Index | データソース名 | 種別 | 取得日時 |
|-------|-------------|------|---------|
| [S1]  | #sales-all 会話 | Slack | 2025-03-07 |
| [T1]  | Sales_Pipeline | Tableau DS | 2025-03-07 |
| [T2]  | Revenue_Dashboard > Monthly View | Tableau View | 2025-03-07 |
| [W1]  | 業界レポート記事タイトル | Web | 2025-03-07 |
```

各ファクトには対応するIndexを付ける:

```
- 売上が前月比15%減少 [T1]
- チームAの案件停滞がSlackで議論されていた [S1]
- 業界全体で同様の傾向 [W1]
```

### Step 7: 分析レポート作成 → ユーザーに報告

以下の構成でレポートをまとめ、**ユーザーに提示する**:

```markdown
# [分析テーマ] 分析レポート

## サマリー

[3行以内の要約]

## 検証した仮説と結果

### 仮説1: [タイトル]

- **判定**: 支持 / 棄却 / 一部支持
- **根拠**: [具体的データ引用 + Index]
- **補足**: [Web情報等の外部コンテキスト + Index]

### 仮説2: ...

## データソースIndex

[Step 6のIndex表]

## 推奨アクション

1. [具体的な次のステップ]
2. ...
```

レポート提示後、`ask_user_input_v0` ツールで**分析フロー図を作成するかどうかの選択ウィジェット**を表示する。

- **ここで必ずレスポンスを終了し、ユーザーの返答を待つ**
- ユーザーが「作成する」を選択した場合 → [references/flow-diagram-spec.md](references/flow-diagram-spec.md) の仕様に従いSVGフロー図を生成・表示する
- ユーザーが「作成しない」を選択した場合 → フロー図をスキップし、異論や追加分析の要望があるか確認する

> ⚠️ HARD STOP: レポート提示後は必ずターンを終了すること。ユーザーの次のメッセージを受け取るまでフロー図生成やStep 8を開始しない。

### Step 8: Slack Canvas作成 → チャンネルに共有

ユーザーが分析結果を承認したら:

1. `slack_search_channels`で投稿可能なチャンネル一覧を取得し、ユーザーに提示
2. ユーザーが共有先チャンネルを選択するまで待つ（複数選択可）
3. 選択されたチャンネルごとに:
   - `slack_create_canvas`でレポートをCanvas化
     - `title`: "[分析テーマ] 分析レポート - YYYY/MM/DD"
     - `content`: レポート本文（Markdown）
     - `channel_id`: 選択されたチャンネルのID
   - `slack_send_message`でCanvasのリンクと要約をチャンネルに投稿

投稿メッセージのフォーマット:

```
[分析テーマ]の分析レポートを作成しました。
[Canvas リンク]

要点:
- [サマリー1行目]
- [サマリー2行目]
```

## 中断・例外ハンドリング

- **MCP認証エラー**: Slack MCP または Tableau MCP で認証エラー（401, 403, トークン期限切れ等）が発生した場合、**ワークフローを一時停止**し、ユーザーに以下を伝える:
  1. どのMCPで認証エラーが発生したか
  2. エラーメッセージの内容
  3. 「認証を修正して、準備ができたら教えてください。続きから再開します。」
  - 認証エラーが解消されるまで該当MCPのツールを再呼び出ししない。ユーザーから再開の合図があったら、失敗したステップから続行する
- **Slack MCP未接続**: Step 1をスキップし、ユーザーに直接仮説を出してもらう。Step 8もスキップ
- **Tableau MCP未接続**: Step 3-4をスキップし、ユーザーにデータを提供してもらう
- **データ不足**: 追加で必要なデータをユーザーに確認してから進む
- **仮説が全て棄却**: 新しい仮説を生成するかユーザーに確認
